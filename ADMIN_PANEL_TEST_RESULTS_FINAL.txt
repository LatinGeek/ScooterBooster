ISSUE #1: SENTRY INSTRUMENTATION FAILURE
---
Severity: 🔴 CRITICAL
Location: Build process / Production deployment
Impact: Error tracking completely broken, cannot monitor production
Files: All pages (affects error tracking system-wide)

Symptom:
  Build fails with: "EPERM: operation not permitted, unlink .next/BUILD_ID"
  Root cause: Missing or inaccessible require-in-the-middle dependency

Fix:
  Option A (Recommended):
    pkill -f next
    rm -rf .next node_modules
    npm install
    npm run build
    npm start

  Option B (Temporary):
    SENTRY_SUPPRESS_INIT_WARNINGS=true npm run dev

Time: 15-20 minutes
Risk: NONE - safe to rebuild

ISSUE #2: DASHBOARD TREND CALCULATION BUG
---
Severity: 🟠 HIGH
Location: /admin page (Dashboard Overview)
File: src/app/admin/page.tsx (lines 91-123)
Impact: 30-day trend metrics inaccurate, business decisions based on bad data

Problem:
  - Bookings with malformed createdAt field are silently excluded
  - No error logging when exclusion occurs
  - If data quality degrades, admin doesn't know

Example:
  Real data: 100 bookings created yesterday
  Dashboard shows: 87 bookings (13 silently skipped)
  Admin thinks: Business is slower than it is

Fix:
  Add robust date extraction function with proper error handling
  Check for: string format, Firebase Timestamp, Date object
  Log warnings when format is unknown
  Use extracted date safely in trend calculation

Time: 10-15 minutes
Risk: LOW - purely additive fix

ISSUE #3: NO INPUT VALIDATION ON SETTINGS PAGE
---
Severity: 🟡 MEDIUM
Location: /admin/settings page
File: src/app/admin/settings/page.tsx
Impact: Invalid fee values break platform pricing

Problem:
  - Service fee can be set to negative values (breaks pricing)
  - No maximum bound (could be 999999999)
  - No type checking (could be float or string)
  - No user feedback if validation fails

Example:
  Admin sets fee to -100
  System now CREDITS users instead of charging them
  Result: Financial loss

Fix:
  Add validation before API call:
  - Check fee >= 0
  - Check fee <= 100000
  - Check fee is integer
  - Show error messages in Spanish

Additional fix:
  - "Configuracion" → "Configuración" (add accent)
  - "Comision" → "Comisión" (add accent)

Time: 10 minutes total
Risk: NONE - purely defensive

ISSUE #4: MISSING ERROR BOUNDARIES
---
Severity: 🟡 MEDIUM
Location: Multiple pages
Files: technicians/page.tsx, users/page.tsx, bookings-client.tsx, reviews/page.tsx
Impact: Page crashes if data is malformed, admin loses access to features

Problem:
  - No try-catch when rendering data
  - No fallback UI if data parsing fails
  - Admin can't access page during data issues
  - Entire page unusable if single record is corrupted

Example:
  One technician record has missing fields
  Users page tries to render all technicians
  Page crashes completely
  Admin can't manage users, technicians, anything

Fix:
  Option A: Wrap data access in try-catch
    Safe defaults for missing fields
    Error message shown for bad records
    Rest of page still works

  Option B: Create ErrorBoundary React component
    Catches rendering errors
    Shows fallback UI
    Logs errors for debugging

Time: 20-30 minutes
Risk: NONE - purely defensive

---

ISSUE #5: HARD LIMIT ON USER PAGINATION
---
Severity: 🟡 MEDIUM (becomes CRITICAL in ~3 months)
Location: /admin/users page
Files: src/app/admin/users/page.tsx + /api/admin/users/route.ts
Impact: Cannot access all users as system grows, user management impossible

Problem:
  - API hard-coded to return max 150 users
  - No pagination controls
  - No "next page" button
  - As user count grows, older users become invisible

Timeline to Critical:
  Current growth: ~50 users/month
  Hard limit: 150 users
  Time to hit limit: ~3 months

Example:
  Today: 75 users, can see all
  In 3 months: 225 users, can only see 150 most recent
  In 6 months: 375 users, can only see 150 most recent
  Old users: Invisible to admin, can't enforce policies

Fix:
  Implement cursor-based pagination:
  - Get pageSize (25) + 1 record from database
  - Return first pageSize records + cursor to next page
  - Add Next/Previous buttons to UI
  - Allow browsing through all users

Time: 30-45 minutes
Risk: MEDIUM - requires API and UI changes, but low risk if done correctly

---

ISSUE #6: NO CONFIRMATION ON DESTRUCTIVE ACTIONS
---
Severity: 🟡 MEDIUM
Location: Multiple pages
Files: bookings-client.tsx, scooters-client.tsx, services-client.tsx, users/page.tsx
Impact: Accidental data loss, no undo capability

Problem:
  - Delete/cancel operations execute immediately
  - No confirmation dialog
  - High risk of accidental clicks
  - No recovery without database restore

Affected Actions:
  - Cancel booking (immediate)
  - Delete brand (immediate)
  - Delete model (immediate)
  - Delete service (immediate)
  - Delete user (immediate)

Example:
  Admin clicks "Cancel" on wrong booking by accident
  Booking cancelled immediately
  No undo
  Customer affected
  No way to recover

Fix:
  Simple option: window.confirm() dialog
    if (!window.confirm("Are you sure?")) return
    // proceed with action

  Better option: Modal dialog component
    Better UX, can be styled
    More professional

Time: 15-20 minutes
Risk: NONE - purely additive

ISSUE #8: MISSING TIMEZONE AWARENESS
---
Severity: 🟢 LOW
Location: /admin/audit page
File: src/app/admin/audit/page.tsx
Impact: Minor confusion, timestamps hard to read

Problem:
  - Audit log shows UTC timestamps: "2026-05-05T16:30:00Z"
  - Admin in EST (UTC-4) sees different time than action occurred
  - Hard to correlate events with real-world actions

Example:
  Actual action time: 12:30 PM EST (admin's timezone)
  Audit log shows: 16:30 UTC
  Admin thinks: "When did this happen?" (confusing)

Fix:
  Format timestamps to admin's local timezone
  Show: "05/05/2026 12:30 p.m. EDT" instead of "2026-05-05T16:30:00Z"
  Use Intl.DateTimeFormat API

Time: 10-15 minutes
Risk: NONE - purely UI improvement
